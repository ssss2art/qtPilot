Protocol Reference

How MCP Servers Work

Model Context Protocol — connecting AI models to external tools and data sources over a standardized JSON-RPC interface

Architecture

🖥️
Host Application
Claude Code, Claude Desktop, or any app embedding an LLM. Creates and manages MCP clients. Decides which servers to connect to.
MCP Client
JSON-RPC 2.0 stdio / SSE
⚙️
MCP Server
Lightweight process exposing capabilities — tools, resources, and prompts — through the standardized MCP protocol.
Tools Resources Prompts
native API SDK / HTTP
🗄️
External Resources
Databases, APIs, filesystems, or services the server wraps. Includes GitHub, Postgres, Slack, browser automation, etc.
Files APIs DBs
Transport Options → stdio (subprocess) SSE (HTTP stream) JSON-RPC 2.0 request / response / notification stateful sessions

Server Capabilities

🔧
Tools
Functions the model can call — run a shell command, query a DB, call a REST API, write a file. Each tool has a name, description, and JSON Schema for input validation.
📄
Resources
Read-only data sources the model can reference — files, database rows, API responses. Identified by URI, streamed as text or binary.
💬
Prompts
Server-defined reusable prompt templates with typed arguments. Unlike tools (which do things) or resources (which expose data), prompts are pre-written instructions the server injects into the conversation — e.g. "summarise this issue", "review this diff". The server fills in argument values and returns ready-to-use messages for the model.

Message Examples

Initialization Session startup — both sides declare who they are and what they support before any work begins.
initialize Request →
Client opens a session, declares protocol version and its own capabilities.
// Client → Server
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "protocolVersion": "2024-11-05",
    "clientInfo": {
      "name": "claude-code",
      "version": "1.0.0"
    },
    "capabilities": {
      "sampling": {}
    }
  }
}
initialized Notification ←
Server confirms initialization. No id field — notifications never get a reply.
// Server → Client (no reply needed)
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}

// Server advertises what it supports
// in the initialize response:
{
  "capabilities": {
    "tools": { "listChanged": true },
    "resources": {},
    "prompts": {}
  }
}
Tools Discover available functions, then invoke one. The model drives the call; the server executes it.
tools/list Request →
Client asks what tools the server exposes. Server returns schemas Claude can use to call them.
// Client → Server
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/list"
}

// Server → Client
{
  "result": {
    "tools": [{
      "name": "query_database",
      "description": "Run a SQL query",
      "inputSchema": {
        "type": "object",
        "properties": {
          "sql": { "type": "string" }
        }
      }
    }]
  }
}
tools/call Request →
Model decides to call a tool. Client forwards the call to the server and returns the result to the model.
// Client → Server
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tools/call",
  "params": {
    "name": "query_database",
    "arguments": {
      "sql": "SELECT * FROM users LIMIT 5"
    }
  }
}

// Server → Client
{
  "result": {
    "content": [{
      "type": "text",
      "text": "id | name\n1  | Alice\n2  | Bob"
    }],
    "isError": false
  }
}
Resources Read-only data the model can pull into context — files, database snapshots, API responses — addressed by URI.
resources/list Request →
Client asks what resources are available. Server returns URIs with MIME types so the model knows what it can read.
// Client → Server
{
  "jsonrpc": "2.0",
  "id": 4,
  "method": "resources/list"
}

// Server → Client
{
  "result": {
    "resources": [{
      "uri": "file:///project/README.md",
      "name": "README",
      "mimeType": "text/markdown"
    }, {
      "uri": "db://users/schema",
      "name": "Users table schema",
      "mimeType": "application/json"
    }]
  }
}
resources/read Response ←
Client fetches a resource by URI. Server streams the content back as text or binary.
// Client → Server
{
  "jsonrpc": "2.0",
  "id": 5,
  "method": "resources/read",
  "params": {
    "uri": "file:///project/README.md"
  }
}

// Server → Client
{
  "result": {
    "contents": [{
      "uri": "file:///project/README.md",
      "mimeType": "text/markdown",
      "text": "# My Project\n..."
    }]
  }
}
Prompts Server-owned instruction templates. The server authors the wording; the client just supplies arguments and gets back ready-to-use messages.
prompts/list Request →
Client asks what reusable prompt templates the server offers. Each prompt declares its typed arguments.
// Client → Server
{
  "jsonrpc": "2.0",
  "id": 6,
  "method": "prompts/list"
}

// Server → Client
{
  "result": {
    "prompts": [{
      "name": "summarize_issue",
      "description": "Summarise a GitHub issue",
      "arguments": [{
        "name": "issue_number",
        "description": "Issue # to fetch",
        "required": true
      }]
    }]
  }
}
prompts/get Response ←
Client retrieves a filled-in template. Server injects the argument values and returns ready-to-use messages for the model.
// Client → Server
{
  "jsonrpc": "2.0",
  "id": 7,
  "method": "prompts/get",
  "params": {
    "name": "summarize_issue",
    "arguments": {
      "issue_number": "42"
    }
  }
}

// Server → Client
{
  "result": {
    "messages": [{
      "role": "user",
      "content": {
        "type": "text",
        "text": "Summarise issue #42: ..."
      }
    }]
  }
}
Progress & Notifications Fire-and-forget messages — no id, no reply expected. Used for progress, logs, and change events.
notifications/progress Notification ←
Server reports progress on a long-running operation that was started with a progressToken.
// Server → Client
{
  "jsonrpc": "2.0",
  "method": "notifications/progress",
  "params": {
    "progressToken": "op-42",
    "progress": 60,
    "total": 100
  }
}

// Server → Client (log line)
{
  "method": "notifications/message",
  "params": {
    "level": "info",
    "data": "Processing row 600/1000"
  }
}
notifications/resources/updated Notification ←
Server tells the client that a resource has changed so the client can re-read it or invalidate its cache.
// Server → Client (resource changed)
{
  "jsonrpc": "2.0",
  "method": "notifications/resources/updated",
  "params": {
    "uri": "file:///project/README.md"
  }
}

// Server → Client (tool list changed)
{
  "method": "notifications/tools/list_changed"
}

// No id field on any notification —
// sender never waits for a reply.

Session Lifecycle

1
Connect
Host spawns the server as a subprocess (stdio) or connects to a remote URL (SSE). A persistent, stateful channel is established.
2
Negotiate capabilities
initialize → initialized handshake. Both sides declare protocol version and what they support (tools, resources, prompts, sampling).
3
Discover
Client calls tools/list, resources/list, and prompts/list. Results are injected into the model's context so it knows what's available.
4
Execute
Model decides to call a tool. Client sends tools/call. Server executes against the real resource and returns content. Model receives the result and continues reasoning.
5
Notify
Either side fires one-way notifications/* messages — progress, logs, resource changes. No reply expected or sent.
6
Shutdown
Client sends shutdown request. Server cleans up and exits. For stdio servers, the host can also simply close the pipe.

Probe Injection

Windows DLL Injection

  1. qtPilot-launcher.exe sets up DLL search path
  2. Launches target app with qtpilot.dll in load path
  3. DllMain auto-initializes the probe
  4. WebSocket server starts on port 9222
  5. Optional: --inject-children hooks CreateProcessW via Detours

Linux LD_PRELOAD

  1. LD_PRELOAD=libqtpilot.so ./myapp
  2. Shared library loaded before app starts
  3. Defers init until QCoreApplication exists
  4. WebSocket server starts on port 9222
  5. LD_PRELOAD auto-propagates to child processes

qtpilot.dll / libqtpilot.so — Probe
▸ Core — Object Tracking ObjectRegistry · ObjectResolver · SymbolicNameMap
ObjectRegistry
Tracks every QObject via lifecycle hooks. Maintains QObject* ↔ hierarchicalId maps. Generates IDs like MainWindow/central/btn. Uses QRecursiveMutex for thread safety.
ObjectResolver
Three-tier ID resolution: #N numeric → symbolic → hierarchical. First match wins. Numeric IDs cleared on client disconnect.
SymbolicNameMap
Friendly name aliases. Register "submit" → "MainWindow/.../QPushButton". Persists to JSON. Auto-loads from env or default file.
▸ Introspection — Reading & Observing MetaInspector · SignalMonitor · EventCapture · A11yTreeWalker · +2
MetaInspector
Reads QMetaObject for properties, methods, signals, class hierarchy. Invokes slots and reads/writes property values.
SignalMonitor
Dynamic signal subscriptions via SignalRelay helper objects. Connects to any signal at runtime and pushes emission notifications.
EventCapture
Global event filter on QCoreApplication. Captures mouse, keyboard, focus, window events and pushes them as notifications.
A11yTreeWalker
Builds accessibility tree with numbered refs for Chrome mode. Maps Qt accessibility roles to Chrome-compatible roles.
ModelNavigator
Inspects QAbstractItemModel instances — row/column counts, role names, paginated data fetching.
QmlInspector
Extracts QML metadata: component id, source file path, QML type name. Bridges the QML/C++ boundary.
▸ API Modes — Method Registration NativeModeApi (33) · ComputerUseApi (13) · ChromeModeApi (8)
NativeModeApi qt.*
33 methods. Full object introspection, property read/write, method invoke, signal subscribe, UI interaction, model/QML access.
ComputerUseApi cu.*
13 methods. Screenshot + coordinate-based mouse/keyboard. No object IDs needed — pure screen interaction.
ChromeModeApi chr.*
8 methods. Accessibility tree with ephemeral numbered refs. Click, form input, navigate by ref ID.
▸ Transport — Wire Protocol JsonRpcHandler · WebSocketServer :9222 · DiscoveryBroadcaster :9221
JsonRpcHandler
Parses JSON-RPC 2.0 messages. Routes method to registered handler lambdas. Formats responses and structured errors (-32601 method not found, etc.).
WebSocketServer
Single-client WebSocket on port 9222. Receives requests, delegates to JsonRpcHandler, sends responses. Also pushes unsolicited notifications (signals, events, lifecycle).
DiscoveryBroadcaster
UDP beacon every 5s on port 9221. Announces PID, port, mode, Qt version.
